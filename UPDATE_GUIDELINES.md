# 程序包更新规范与流程

本文件用于后续更新本程序包时统一执行标准。除非用户另有要求，更新时应优先保证功能稳定、页面加载顺畅、版本记录清楚、用户界面简洁。

## 一、基本更新原则

1. **版本号必须同步调整**  
   每次功能更新、修复或打包发布后，都应同步检查并更新涉及的版本号，包括页面可见版本号、脚本缓存参数、测试脚本中的版本断言，以及必要的资源引用版本。

2. **不要修改最外层 `README.md`**  
   根目录 `README.md` 只保留项目基本说明和结构说明。更新说明、版本记录、实现细节和临时说明不要写入根目录 `README.md`。

3. **changelog 统一放入 `changelog/` 文件夹**  
   每次更新应在 `changelog/` 下新增对应版本的 changelog，例如 `RIES_v11.4.3_CHANGELOG.md`。不要在压缩包最外层散放 changelog 文件。

4. **用户界面保持克制**  
   制作或修改界面时，如果没有额外要求，不要显示大段 AI 风格描述，不要用过度解释性文案堆满界面。页面应以清晰、短句、可操作提示为主。

5. **测试要求按用户指令执行**  
   如果用户要求对本次更新的模块或功能进行测试，应添加或更新必要的测试脚本，并实际运行相关测试。测试结果应在交付说明中简要列出。

6. **网页必须能正常加载且尽量不卡顿**  
   任何新增的大型数据包、复杂计算、长时间搜索或预计算，都应优先考虑懒加载、分批执行、进度提示、取消机制和首屏减负，避免页面一进入就长时间无响应。

7. **大型资源优先懒加载**  
   体积较大的数据库、预计算表、扩展包、可选模块，不应默认随首页全部加载。只有当用户启用相关功能或输入满足调用条件时才加载。

8. **加载状态必须可见**  
   当页面正在加载某个包、数据库或耗时模块时，应显示明确状态，例如“正在加载某某包”，并尽可能显示进度。已有动画控件可以复用，但必须配合清楚的文字提示。

9. **功能模块应独立接入排序/展示系统**  
   新增模块不应简单混入旧模块名称中。若结果来自独立算法或数据库，应保留模块身份，并按当前排序规则参与综合展示。

10. **解释信息要完整但不要冗长**  
    公式、变量、函数、参数、误差与接受条件应说明清楚。解释应能帮助用户理解结果来源，但不应写成大段论文式描述。

11. **避免破坏旧功能**  
    修改某个模块时，应注意对相关调用链、导出测试钩子、UI 状态、缓存参数、排序逻辑和错误处理的影响。不要为了一个新增功能牺牲已有稳定功能。

12. **保持文件结构清晰**  
    新增资源应放入合适目录，例如脚本和数据放 `assets/`，测试放 `tools/`，版本记录放 `changelog/`。不要在根目录堆放临时脚本、日志或中间文件。

13. **交付前清理临时内容**  
    打包前应删除调试输出、临时代码、无用副本、缓存文件和编辑器生成文件。保留的测试脚本应可重复运行。

14. **错误处理应对用户友好**  
    如果资源加载失败、浏览器不支持某能力或计算被取消，应给出明确提示，而不是让界面静默失败或永久显示加载状态。

15. **更新说明应说明“改了什么”和“为什么”**  
    changelog 不只写文件名变化，还应简要说明功能影响，例如“首屏改为懒加载以降低进入页面时的等待时间”。

## 二、推荐更新流程

1. **确认用户需求**  
   先列出本次需要修改的模块、界面、性能、文档和测试要求。若用户已经给出明确要求，不要重复询问。

2. **建立工作副本**  
   在独立目录中解压并修改程序包，避免直接破坏原始压缩包。

3. **定位相关代码**  
   查找版本号、页面入口、资源加载、目标模块逻辑、排序逻辑、UI 显示逻辑、测试脚本和 changelog 位置。

4. **先做最小可行修改**  
   优先用最小范围修复核心问题，避免无关重构。只有在性能、稳定性或需求明确要求时才重构。

5. **处理版本与缓存参数**  
   完成代码修改后，统一检查页面可见版本号、脚本查询参数、测试断言和资源说明是否需要更新。

6. **补充 changelog**  
   在 `changelog/` 中新增本版本 changelog，记录主要变化、性能调整、测试变化和兼容性注意点。

7. **维护更新规范文件**  
   如更新流程或项目约定发生变化，应同步修订本文件；否则不必反复改动。

8. **运行语法检查**  
   对主要 JavaScript 文件运行语法检查，例如 `node --check ries-script.js` 和必要的入口脚本检查。

9. **运行相关测试**  
   至少运行与本次变更直接相关的 smoke test。若改动排序、懒加载、数据库或 UI 状态，应覆盖这些点。

10. **检查首屏加载**  
    确认首页不再同步加载不必要的大包；需要加载包时，应有清楚的加载提示和进度反馈。

11. **检查文件结构**  
    确认根目录没有散落 changelog、临时文件或测试输出；确认 `README.md` 没有被误改。

12. **重新打包**  
    从工作目录根部重新压缩，保证解压后的目录结构与原包一致。

13. **交付说明**  
    交付时简要说明本次改动点、已运行测试、未改动/无法验证的内容，并提供新的压缩包。

## 三、交付前检查清单

- [ ] 页面可见版本号已更新。
- [ ] 脚本缓存参数已更新。
- [ ] 根目录 `README.md` 未被加入更新说明。
- [ ] 新 changelog 已放入 `changelog/`。
- [ ] 根目录没有散落 changelog。
- [ ] 大型可选资源没有阻塞首屏加载。
- [ ] 加载大型包时有明确状态和进度提示。
- [ ] 新模块已按独立模块身份接入排序或展示系统。
- [ ] 变量、函数和公式解释足够清楚。
- [ ] 相关测试脚本已更新并运行通过。
- [ ] 打包前已清理临时文件。

## 四、Parameters 面板更新规则

1. **默认行为必须兼容**  
   新增或调整 Parameters 控件时，默认值必须保持用户不调整参数时的搜索行为不变；若需求明确要求改变默认搜索范围，应在 changelog 中单独说明。

2. **新模块应暴露合理接口**  
   设计新搜索模块时，应优先把安全、稳定、可理解的开关暴露到 Parameters，例如模块总开关、主要子阶段开关、候选数量、时间预算、搜索高度或深度。内部精度、数值稳定性阈值和容易导致误用的实现细节不应随意暴露。

3. **控件必须真实接入搜索链路**  
   Parameters 中出现的控件必须写入 `readSettings()`，并进入缓存键、模块入口或具体搜索阶段。不能只做 UI 展示而不影响结果。

4. **模块总开关应控制折叠状态**  
   每个搜索模块应有总开关；关闭后折叠或弱化其细分选项，重新打开后恢复可见，避免参数面板过长且难以阅读。

5. **候选数与时间预算应可调但有边界**  
   对用户暴露的 candidate 数量和 stage time limit 应设置合理最小值、最大值与默认值，避免误设导致浏览器长时间卡死。

6. **缓存键必须包含参数变化**  
   如果某个控件会影响搜索结果、搜索范围或输出数量，应同步加入 solve/integer cache key，避免用户调整参数后拿到旧缓存结果。
